# Design Self-Review Checklist

方案设计完成后，逐项回答以下 14 个维度。每项写：✅ 通过 / ⚠️ 发现问题 / ❌ 未覆盖。

---

## 1. Problem
- 要解决的核心问题是什么？
- 不解决什么？
- 这个问题是否真实存在？证据是什么？

## 2. Current State
- 当前实现/系统现状是什么？
- 关键现状证据：
- 相关源码/配置/文档锚点：

## 3. External Facts
- 依赖了哪些外部标准、协议、平台、库行为？
- 每条外部事实的权威来源：
- 这些事实是否可能已过时？

> **[!IMPORTANT]**
> 每条外部事实必须有可验证锚点（文件路径+行号 或 URL+日期）。
> 本次 Phase 1-4 实战中，F5（callTool 是否支持 signal）就因为锚点缺失被用户一击驳斥。

## 4. Core Decisions
- 本方案必须先钉死的决策有哪些？
- 每个决策的备选方案是什么？
- 为什么选当前方案而不是其他方案？

## 5. Architecture Boundaries
- 是否跨越了原有模块/服务/运行时边界？
- 是否污染了不该受影响的环境或入口？
- 依赖方向是否清晰？
- 是否会引入循环依赖或层级倒挂？

## 6. Main Execution Path
- 新能力是否真的进入主链路？
- 哪些路径会消费它？
- 是否只是新增了 API/配置，但没有真正生效？

> **[!IMPORTANT]**
> 用 grep/trace 证明消费路径存在，禁止"类型定义了即能力生效"假设。

## 7. Lifecycle & Ownership
- 谁创建？
- 谁初始化/连接？
- 谁缓存？
- 谁更新？
- 谁释放？
- 失败和重试由谁负责？

## 8. Data & Type Semantics
- 类型定义和运行时语义是否一致？
- 字段名是否表达真实语义？
- 是否存在"类型允许，实际不支持"的情况？

## 9. Compatibility
- 哪些是纯增量？
- 哪些是行为变化？
- 哪些旧用法会失效或变得危险？
- 迁移路径是否明确？

## 10. Security
- 供应链风险是什么？
- 运行时风险是什么？
- 输入/数据/指令混淆风险是什么？
- 权限边界是什么？
- 每类风险的防御机制是什么？

## 11. Validation
- 单元测试怎么做？
- 集成测试怎么做？
- CI 能否封闭跑通？
- 是否依赖外部系统/网络/真实服务？
- 手动验证场景有哪些？

## 12. Rollout
- 分几个阶段？
- 每阶段前置条件是什么？
- 哪些点必须先完成才能进入下一阶段？

## 13. Open Questions
- 还没决定的点有哪些？
- 哪些未决问题会阻断实施？

## 14. Kill Shot
- 如果今天开始做，最可能先卡死的 3 个点是什么？
- 如果上线后出事故，最可能是哪里？
